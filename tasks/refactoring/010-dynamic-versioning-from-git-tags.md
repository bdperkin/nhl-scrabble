# Dynamic Versioning from Git Tags

**GitHub Issue**: #222 - https://github.com/bdperkin/nhl-scrabble/issues/222

## Priority

**LOW** - Nice to Have (Next Quarter)

## Estimated Effort

2-4 hours

## Description

Implement dynamic versioning using Git tags and VCS (Version Control System) integration at build time, eliminating the need for manual version updates in `pyproject.toml`. This automates version management, ensures consistency between Git tags and package versions, and reduces manual maintenance burden.

## Current State

**Manual Version Management:**

```toml
# pyproject.toml (line 21)
[project]
name = "nhl-scrabble"
version = "2.0.0"  # ← Manually updated on every release
```

**Current Workflow:**

1. Decide on next version number
1. Manually edit `pyproject.toml` to update version
1. Commit version bump: `git commit -m "chore: bump version to 2.0.0"`
1. Create Git tag: `git tag v2.0.0`
1. Push changes and tag: `git push && git push --tags`
1. Build and release

**Problems:**

1. **Manual Updates**: Requires remembering to update version in `pyproject.toml`
1. **Inconsistency Risk**: Version in file can diverge from Git tag
1. **Extra Commits**: Version bump commits clutter git history
1. **Error-Prone**: Easy to forget or mistype version
1. **No Dev Versions**: Can't automatically differentiate between released and development versions

**Current State Example:**

```bash
# Current workflow requires manual coordination
$ cat pyproject.toml | grep version
version = "2.0.0"

$ git tag | tail -1
v2.0.0

# If they diverge, it's a problem:
# pyproject.toml: 2.0.0
# git tag: v1.9.0
# Which is correct?
```

## Proposed Solution

### Use hatch-vcs for Dynamic Versioning

Since the project already uses `hatchling` as the build backend, `hatch-vcs` is the natural choice for VCS-based versioning.

**Benefits:**

- Native hatchling integration
- Automatic version from Git tags
- Development version support (e.g., `2.0.1.dev5+g1234567`)
- PEP 440 compliant versions
- Zero manual version maintenance

### Implementation Architecture

```
Git Repository
     ↓
  Git Tag (v2.0.0)
     ↓
hatch-vcs (build time)
     ↓
Dynamic Version
     ↓
Package Metadata (__version__)
```

### Configuration

**pyproject.toml Changes:**

```toml
# Before:
[project]
name = "nhl-scrabble"
version = "2.0.0"  # Manual static version

# After:
[project]
name = "nhl-scrabble"
dynamic = ["version"]  # Version determined dynamically

[tool.hatch.version]
source = "vcs"  # Use version control system

[tool.hatch.build.hooks.vcs]
version-file = "src/nhl_scrabble/_version.py"  # Generated version file

[build-system]
requires = ["hatchling", "hatch-vcs"]
build-backend = "hatchling.build"
```

### Version File Generation

**Auto-generated `src/nhl_scrabble/_version.py`:**

```python
# This file is automatically generated by hatch-vcs
# Do not edit manually
__version__ = "2.0.0"
__version_tuple__ = (2, 0, 0)
```

**Updated `src/nhl_scrabble/__init__.py`:**

```python
# Before:
__version__ = "2.0.0"  # Manually maintained

# After:
try:
    from nhl_scrabble._version import __version__
except ImportError:
    # Fallback for development without build
    __version__ = "0.0.0+unknown"
```

### Versioning Strategy

**Semantic Versioning with Git Tags:**

```bash
# Major release
git tag v3.0.0
# Version: 3.0.0

# Minor release
git tag v2.1.0
# Version: 2.1.0

# Patch release
git tag v2.0.1
# Version: 2.0.1

# Pre-release
git tag v2.1.0-rc1
# Version: 2.1.0rc1

# Development (no tag, 5 commits after v2.0.0)
# Version: 2.0.1.dev5+g1234567
```

**Version Number Components:**

```
2.0.1.dev5+g1234567
│ │ │ │   │ │
│ │ │ │   │ └─ Git commit hash (short)
│ │ │ │   └─── Local version identifier
│ │ │ └─────── Dev version (commits since last tag)
│ │ └───────── Patch version
│ └─────────── Minor version
└───────────── Major version
```


### CI/CD Integration

**GitHub Actions (Build):**

```yaml
# .github/workflows/build.yml
- name: Build package
  run: |
    python -m build

- name: Display version
  run: |
    python -c "from nhl_scrabble import __version__; print(f'Built version: {__version__}')"
```

**GitHub Actions (Release):**

```yaml
# .github/workflows/release.yml
on:
  push:
    tags:
      - 'v*'  # Trigger on version tags

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Required for hatch-vcs

      - name: Build package
        run: python -m build

      - name: Publish to PyPI
        uses: pypa/gh-action-pypi-publish@release/v1
        with:
          password: ${{ secrets.PYPI_API_TOKEN }}
```

**Important**: `fetch-depth: 0` is required to get full git history for version calculation.

## Additional Notes

### Why hatch-vcs?

**Alternatives Considered:**

1. **setuptools-scm**: Popular but requires setuptools backend
1. **hatch-vcs**: Native hatchling integration (project already uses hatchling)
1. **poetry-dynamic-versioning**: For poetry projects only
1. **Manual script**: Custom solution, more maintenance

**Chosen**: `hatch-vcs` because:

- Project already uses hatchling
- Native integration, no backend change
- Simple configuration
- PEP 440 compliant
- Well-maintained

### Version File Strategy

**Options:**

1. **No version file**: Import from package metadata (slower)
1. **Generated version file**: Fast import, build-time generation
1. **Committed version file**: Manual maintenance (defeats purpose)

**Chosen**: Generated version file (`_version.py`) because:

- Fast imports (no pkg_resources lookup)
- Automatically updated on build
- Clean separation of concerns

### Git Tag Format

**Recommended Format**: `vX.Y.Z`

```bash
# Good:
v2.0.0
v2.1.0-rc1
v2.0.1

# Also works but less standard:
2.0.0  # No 'v' prefix
release-2.0.0  # Custom prefix (requires configuration)
```

**Configuration for Custom Prefix:**

```toml
[tool.hatch.version]
source = "vcs"
tag-pattern = "release-(?P<version>.*)"  # Custom pattern
```

### Development Version Semantics

**PEP 440 Development Versions:**

```
2.0.1.dev5+g1234567
     │    │  │
     │    │  └─ Local version (git hash)
     │    └──── Development release number
     └───────── Next version number
```

**Interpretation:**

- `2.0.0` is the last released version
- `2.0.1` will be the next version
- `.dev5` means 5 commits since last tag
- `+g1234567` is the commit hash

### Release Workflow

**Standard Release Process:**

```bash
# 1. Ensure main is clean and tests pass
git checkout main
git pull origin main
make test

# 2. Create annotated tag
git tag -a v2.0.0 -m "Release version 2.0.0"

# 3. Push tag (triggers CI release)

# 4. CI automatically:
#    - Builds package with version 2.0.0

#    - Publishes to PyPI (if configured)

# 5. Verify release

```

**Rollback if Needed:**

```bash
# Delete tag locally and remotely
git tag -d v2.0.0
git push origin :refs/tags/v2.0.0

# Delete GitHub release (if created)
gh release delete v2.0.0
```

### Troubleshooting

**Problem**: Version shows as `0.0.0+unknown`

**Causes:**

- No git tags exist
- Shallow clone (missing history)
- Not in git repository
- `hatch-vcs` not installed

**Solutions:**

```bash
# Check git tags
git tag

# Check git depth
git log --oneline | wc -l

# Create initial tag if needed
git tag v0.1.0

# Ensure full clone in CI
fetch-depth: 0
```

**Problem**: Build fails with "version not found"

**Cause**: Missing `hatch-vcs` in build requirements

**Solution:**

```toml
[build-system]
requires = ["hatchling", "hatch-vcs"]  # Ensure hatch-vcs is listed
```

**Problem**: Development version not updating

**Cause**: Cached version file

**Solution:**

```bash
# Clean build artifacts
rm -rf dist/ build/ src/nhl_scrabble/_version.py

# Rebuild
python -m build
```
