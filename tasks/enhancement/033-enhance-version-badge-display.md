**LOW** - Nice to Have (Next Quarter)

## Description

Enhance the version badge strategy in README.md to better align with the dynamic versioning system (task #010) and prepare for future PyPI publication. While a GitHub release badge already exists, it can be improved to provide more accurate version information and be ready for package distribution.

Current badge shows latest release, but we can:
1. Improve badge visibility and clarity
2. Add PyPI version badge (when published)
3. Ensure badge updates automatically with new releases
4. Document version badge update mechanism

- New GitHub release is created (task #032)
- New git tag is pushed
- Package is published to PyPI

## Additional Notes

### Badge Options Comparison

| Option | Pros | Cons | Use Case |
|--------|------|------|----------|
| **A: Separate stable/pre-release** | Clear distinction, shows both | Takes more space | Projects with frequent pre-releases |
| **B: Single badge (include_prereleases)** | Compact, always shows latest | No distinction | Most projects |
| **C: Tag badge** | Shows all tags, not just releases | Includes non-release tags | Tag-focused workflow |

**Recommendation:** Option B (single badge with `include_prereleases`) - simplest and matches current implementation.

### Shields.io Badge Parameters

**Common parameters:**
```
?label=VERSION          # Custom label text
&color=blue            # Badge color (blue, green, orange, red, etc.)
&logo=python           # Add logo icon
&logoColor=white       # Logo color
&style=flat            # Badge style (flat, flat-square, plastic, etc.)
&include_prereleases   # Include pre-release versions
```

**Current badge breakdown:**
```
https://img.shields.io/github/v/release/bdperkin/nhl-scrabble?include_prereleases
│                       │      │                         │
│                       │      │                         └─ Repo name
│                       │      └─ Version type (release)
│                       └─ Badge type (GitHub version)
└─ Shields.io base URL
```

### Auto-Update Mechanism

**Shields.io Caching:**
- Default: 5 minute cache
- Hourly refresh for GitHub API badges
- Can force refresh: append `&cacheSeconds=0` (not recommended)

**Update Flow:**
1. Developer pushes tag: `git push origin v2.1.0`
2. GitHub Actions creates release (task #032)
3. GitHub API updates releases endpoint
4. Shields.io cache expires (~5 min)
5. Badge auto-refreshes from GitHub API
6. README shows new version
